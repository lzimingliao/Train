# 火车票管理系统

## 一、实验整体目标

1. 掌握线性表（数组/链表）、哈希表、树（排序树）等基础数据结构的设计与应用；
2. 熟练运用排序（冒泡/快速排序）、查找（二分查找、哈希查找）等经典算法解决实际问题；
3. 理解业务逻辑到算法模型的转化过程，培养模块化编程和异常处理能力；
4. 逐步掌握复杂场景下的算法优化（如动态分配、冲突处理、路径规划）。

## 二、实验环境与开发要求

1. 开发语言：C/C++、Python、java等；
2. 数据存储：内存数据结构（数组/链表），进阶可引入文件持久化（txt/bin）；
3. 交互方式：GUI；
4. 编码规范：函数模块化、变量命名规范、关键逻辑添加注释、处理边界异常（如空查询、重复身份证）。

## 三、分任务详细实验内容与要求

### 任务 1：基础功能实现

#### 1.1 用户管理模块

**功能要求 ：**

* **数据字段：**姓名（字符串）、身份证号（字符串，主键，不可重复）、手机号码（字符串）；
* **核心操作：**
  * 录入：检查身份证号唯一性，重复则提示错误，录入成功返回用户信息；
  * 修改：通过身份证号定位用户，可修改姓名 / 手机号，身份证号不可改；
  * 删除：通过身份证号删除用户，删除前确认，无该用户则提示；
  * 辅助：可查询所有用户（按身份证号排序），便于测试。

**算法要求 ：**

* **存储结构：**推荐哈希表（以身份证号为 key），或有序数组（二分查找提升查询效率）；
* **异常处理：**空输入、非法身份证号（如长度非 18 位）、重复身份证号的校验。

#### 1.2 车次管理模块

**功能要求 ：**

* **数据字段：**车次号（字符串，主键）、出发站（字符串）、出发时间（日期+时间，如 2026-03-01 08:30）、到达站（字符串）、到达时间（日期+时间）、总车票数量（整数）、剩余车票数量（整数，初始等于总数量）、票价（浮点数）；
* **核心操作：**
  * 录入：检查车次号唯一性，出发/到达时间逻辑校验（到达时间晚于出发时间），录入成功初始化余票；
  * 修改：通过车次号定位，可修改出发/到达站、时间、票价、总票数（修改票数时需同步调整余票，余票≤总票数）；
  * 删除：通过车次号删除，删除前确认，需检查是否有已订票记录（有则禁止删除）。

**算法要求 ：**

* **存储结构：**线性表（数组/链表）或哈希表（车次号为key）；
* **逻辑校验：**出发时间<到达时间、总票数≥ 0、票价≥ 0等边界条件。

#### 1.3 车票查询模块

**子功能 1：区间车次查询**

* **功能：**输入出发日期、出发站、到达站，查询符合条件的车次，返回车次号、出发/到达时间、余票数量、票价；
* **排序要求：**查询结果按 余票数量从多到少 排序（余票相同则按出发时间升序）；
* **算法要求：**实现快速排序/冒泡排序，掌握自定义排序规则（多关键字排序）。

**子功能 2：车站车次查询**

* **功能：**输入日期、车站，查询当天经过该车站的所有车次（包含出发站/到达站为该车站的车次）；
* **排序要求：**按 发车时间从早到晚 排序；
* **算法要求：**顺序查找筛选符合条件的车次，再进行升序排序。

#### 1.4 订票模块

**功能要求 ：**

* **前置条件：**先通过区间/车站查询获取目标车次，确认有余票；
* **核心流程：**
  * 输入用户身份证号，校验用户是否存在（不存在则提示先录入用户信息）；
  * 输入目标车次号，校验该车次余票>0；
  * 生成车票信息：用户姓名、身份证号、车次号、出发日期、出发/到达站、票价、订票时间；
  * 扣减该车次的剩余车票数量（余票-1）；
  * 数据存储：车票信息需单独存储（线性表/哈希表），关联用户和车次。

**算法要求 ：**

* **关联查询：**通过身份证号关联用户、车次号关联车次，保证数据一致性；
* **并发考虑（简化）：**暂不考虑多用户同时订票，重点保证单用户订票时余票扣减的正确性。

#### 任务 1 验收要求

* 所有功能可独立运行，输入合法/非法数据均能正确响应；
* 给出测试用例（如录入3个用户、5个车次，查询区间车次并排序，订票后余票正确扣减）；
* 核心代码注释清晰。

### 任务 2：功能进阶

#### 2.1 用户注册模块

**功能要求 ：**

* **输入要求：**用户名、密码、真实姓名、身份证号（格式校验）、手机号（可选）；
* **校验规则：**
  * 用户名 / 身份证号唯一性校验（禁止重复注册）；
  * 身份证号格式校验（18 位，最后一位支持 X）、手机号格式校验（11 位数字）；
  * 密码强度校验（弱密码如 12345678 需提示，强制修改）；
  * 注册完成：生成唯一用户 ID，返回注册成功提示，需登录后才可使用功能。

**算法要求 ：**

* **唯一性校验：**用户名 / 身份证号采用哈希表存储，查重时间复杂度≤O (1)；

#### 2.2 登录与权限管理模块

**功能要求 ：**

* 增加用户登录界面，区分管理员用户（无须注册、预设）与普通用户；
* 普通用户具有车票查询、订票、退票、改签、个人信息查询等功能；
* 管理员用户具有车票查询、车次管理、用户管理等功能；

**算法要求 ：**

* **权限存储：**采用 “权限树” 结构存储角色与权限的映射关系（根节点：角色，子节点：模块，叶子节点：具体功能）；
* **权限校验：**实现递归 / 非递归遍历权限树算法，快速匹配当前操作对应的权限节点；

#### 2.3 退票模块

**功能要求 ：**

* **查询：**用户使用用户名+密码登录系统，查询该用户所有已订票信息（显示车次、日期、座位号等）；
* **退票：**选择目标车票，确认退票后：
  1. 删除该车票记录；
  2. 恢复对应车次的剩余车票数量（余票 + 1）；
  3. 若车票已改签，需同步清理改签记录。

**算法要求 ：**

* **多条件查询：**姓名+身份证号联合筛选（注意同名情况）；
* **数据一致性：**退票操作需保证车次余票和车票记录的同步更新。

#### 2.4 改签模块

**功能要求 ：**

* **改签规则：**
  1. 同日期同区间改签：改签为出发当天、出发/到达站相同的其他车次；
  2. 同车次不同日期改签：改签为同一车次的其他日期（需该车次该日期有余票）；
* **改签流程：**
  1. 查询用户已订票信息，选择可改签车票；
  2. 校验目标改签车次/日期是否符合规则，且有余票/可用座位；
  3. 扣减目标车次余票，恢复原车次余票；
  4. 更新车票信息（车次/日期/座位号），记录改签时间。

**算法要求 ：**

* **规则校验：**先判断改签类型，再校验区间/车次/日期是否符合要求；
* **冲突处理：**若改签目标车次无票，提示 “改签失败，目标车次余票不足”。

#### 2.5 座位分配模块

**功能要求 ：**

* **扩展车票信息：**增加座位号（如 “01A”“05B”），按车次座位容量（如 80 个座位）分配；
* **选座/派座规则：**
  1. 优先允许用户自选座位（输入座位号，校验未被占用则分配）；
  2. 若用户不选座，按订票时间先后顺序自动分配最小编号可用座位；
  3. 座位占用状态需实时更新，售完则提示 “无可用座位”。

**算法要求 ：**

* **状态管理：**为每个车次维护 “座位状态数组”（布尔型，true =已占用，false =可用）；
* **查找算法：**快速查找可用座位（顺序查找或标记法）。

#### 任务 2 验收要求

* 用户登录与权限管理功能正确，严格区分不同类型用户权限；
* 退票、改签功能严格遵循规则（如跨区间改签提示错误）；
